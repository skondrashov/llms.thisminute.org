What is orchestration?

A single model in a single context window is good at a lot of things, but there are tasks it can’t hold in one go: refactors that span dozens of files, research projects with too many threads, workflows that need a cautious planner and an aggressive implementer and a skeptical reviewer. Orchestration is the set of patterns for splitting work across multiple agents, subagents, phases, or processes, and making sure the pieces fit back together.

None of this is new. Human organizations have been solving the same problem for thousands of years: how do we coordinate a pile of limited minds into something that can tackle a problem bigger than any one of them? A military command structure, a newsroom, an ant colony, a jazz ensemble, a hospital on a busy night. These are all orchestration patterns, and the LLM version inherits most of the vocabulary.

The primitives

Almost every pattern in the catalog is built out of the same small set of primitives.

  • Roles. Distinct agent personas with their own system prompts, tool permissions, and scope. A planner, an executor, a reviewer, a router.
  • Skills. Reusable instruction sets that get loaded into context on demand. A security review checklist, a migration playbook, a style guide. Unlike roles (which frame identity), skills frame capability for a specific task and then leave.
  • Context boundaries. Who sees what. Subagents usually start with a minimal context and return a summary so the parent’s window doesn’t bloat. This is the single most important design choice in any multi-agent system.
  • Coordination channels. How agents talk to each other. A supervisor dispatching tasks, a shared task list, a mailbox, an artifact (a spec document, a diff, a plan) that multiple agents read and write.
  • Gates. Places where the flow pauses for approval: a human, a judge agent, a test suite, a compile step. Gates are where orchestration meets governance.

Swap out any of these and you get a different pattern. The catalog below is essentially a combinatorial space of those primitives, dressed up with domain-specific names.

Roles and skills are the same mechanism

If you look at a role file and a skill file side by side, they’re both just text that ends up in the context window. A role says “you are a code reviewer who cares about security and readability.” A skill says “here is how to do a dependency audit: check lockfiles, verify hashes, flag unmaintained packages.” The model sees both as tokens. It doesn’t know one is supposed to be identity and the other is supposed to be capability.

The practical difference is lifecycle. A role stays in context for an entire session or agent lifetime. A skill gets injected when needed and doesn’t persist. In Claude Code, your CLAUDE.md is a role (always loaded). A slash command like /review is a skill (loaded on demand, shapes one task, then it’s just history). In LangGraph or CrewAI, you see the same split: agent definitions are roles, tool descriptions and few-shot examples are skills.

This matters for orchestration because loading the right context at the right time is the whole game. An agent with 15 skills loaded permanently is wasting window space on 14 skills it doesn’t need right now. An agent with no skills has to figure everything out from scratch. The orchestration patterns in the catalog below are, at their core, strategies for managing this: which agents see which instructions, when, and for how long.

The harness is already orchestrating

If you’ve used Claude Code, Cursor, Aider, Cline, or anything similar in the last six months, you’ve already been inside an orchestration system. The Plan mode / Execute mode split is a two-phase pipeline. Subagent dispatch is a supervisor-router. Parallel agents in separate worktrees is best-of-n racing. Hooks that fire on lifecycle events are a stigmergic coordination layer bolted under the conversation.

A big chunk of “how should I orchestrate this?” is really “which of these harness features am I already underusing?” The Core lens below flags, for each pattern, which harnesses ship a native version of it. If a pattern looks useful, that row tells you where to go enable it.

Rule of thumb: if a pattern ships natively in a harness you already use, try that first. Hand-rolling an orchestration pattern is usually ten times more work than flipping a config.

How to read the catalog

The catalog is split into three lenses, filterable at the top:

  • Core. Patterns every agent developer will meet. Industry-standard shapes, with the harnesses that ship each one.
  • In the Wild. Patterns from real-world human organizations. NASA mission control, Toyota production, airline crew resource management. Useful when your agent system is starting to look like a small company and you want prior art.
  • The Garden. Nature-inspired, philosophical, and exotic patterns. Ant colonies, pirate democracies, mycelial networks. Worth browsing even if you don’t use any of them directly.

Each pattern card shows a structure (the abstract shape: hub-and-spoke, peer ensemble, assembly pipeline, and so on) and a topology (the edge pattern: orchestrated, mesh, pipeline, etc.). The detail view has roles, forums, a diagram, and for Core patterns a “ships natively in” list that links out to vendor docs.
