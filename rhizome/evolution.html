How These Projects Organized Themselves

Nine projects, three coordination models, ~44 agents — from a 10-daemon research lab that generated 8,753 lines of forum in a single day, to a catalog that works fine with one agent, to a forge that watches them all and extracts what works. What actually happened.

February–March 2026 · Ongoing · Now tracking 9 projects, ~44 agents

Project 1: agi

A composable AI research project — building intelligence from small, specialized transformer models arranged in a directed acyclic graph. Board games as testbeds (TicTacToe, Connect4, Go), later expanding to 20+ domains. The first Claude coding project; the one everything else descended from.

Feb 1, 2026

The 10-Daemon Era

The project launched with 10 agents (called "daemons" at the time): trainer, scientist, architect, tester, moderator, curator, editor, coach, police, orchestrator. The coach tracked zoo membership obsessively (77→84→91→98 models in a day). The police filed enforcement reports every cycle ("Twelfth consecutive cycle with no violations"). The curator maintained count reconciliations.

A forum was set up from day one as the coordination mechanism — threaded discussions with mandatory voting.

orchestrator trainer scientist architect tester moderator curator editor coach police
What broke: Everything was bureaucratic noise. The police, coach, editor, and curator generated enormous overhead (enforcement reports, count reconciliations, daemon file reviews) with no proportional scientific output. Within 24 hours the forum hit 8,753 lines. The 10-daemon system was consolidated to 6 agents that same evening.

The Great Consolidation

Trainer became builder. Scientist+architect became strategist. Tester+moderator became verifier. Curator+editor+coach became librarian. Police was eliminated. "Daemon" became "agent." SCIENCE.md became STRATEGY.md.

orchestrator builder strategist verifier librarian

Then the skeptic was added, prompted by the human asking "Why are we using minimax for Go at all?" The skeptic immediately invalidated multiple claims: all Go results were built on a broken minimax oracle (it mishandled pass moves). "6500% improvement" was cherry-picked. "70% zero-shot transfer" was false. "Universal features" was overstated. The Feynman quote was embedded in its role file: "The first principle is that you must not fool yourself."

Feb 10

The Rebuild

After Go-5x5 AlphaZero training plateaued (42 cycles, 14 cycles of regression), the team abandoned weeks of work and went back to TicTacToe to prove the real architecture. The debugger found 3 bugs (TT corruption causing 23.5% wrong labels, hard single-move labels, value head collapse). After fixing: 100/100 draws vs minimax, then 99.6% optimal on all 4,520 positions.

What surprised: Adding 100% accurate features could HURT performance. Go-5x5 code features (liberty count, capture opportunity) caused a statistically significant regression from 70% to 50%. More information doesn't always help.

The real breakthrough was policy-guided MCTS: the same 160K-parameter model went from 54% to 87% just by using its policy as priors in tree search. Then ONNX export + a Rust MCTS implementation delivered 102x speedup (77 seconds/game to 756 milliseconds).

Five roles modeled after agi's post-consolidation team: builder, tester, strategist, skeptic, orchestrator. LLM extraction via Claude Haiku replaced keyword categorization. Event clustering and narrative synthesis via Sonnet.

The GDELT Crisis

GDELT was sampled at 7%. The underlying dataset grew 10x without anyone noticing, yielding 45,000 stories/day instead of 100. Events bloated to 25,000. Three new roles added: deployer, economist, designer.

Week 2

Domain Specialization

The narrative analyzer only understood geopolitical news. Fix: separate analysis per domain (news, sports, entertainment, positive), each with its own Sonnet prompt, quality criteria, and event caps. A 5th domain (curious/human-interest) followed.

What broke: Sports clustering grouped rugby, cricket, and tennis into one "situation." Fix: "ONE SPORT PER SITUATION" in the prompt. Entertainment regex matched real places — "batman" matched Batman Province, Turkey.
Week 2, day 3

Librarian added as the 9th agent — the forum had hit 1,008 lines in one session. The strategist proposed 13 audience segments and 8 presets. The skeptic tore it apart: numbers inflated 10–100x, 2 of 3 "immediate" items already built, time estimates 2–3x too low.

Project 3: rhizome

This catalog. 205 organizational patterns, 7 curated from real ecosystem usage. One steward agent, no forum, no protocol file. The right amount of structure for a browsable catalog turns out to be almost none.

March, week 2

One Steward

A single agent handles everything: frontend, data curation, API work, build pipeline, accessibility, deploy coordination. It maintains one memory file. There's no forum because there's nobody to discuss things with. No protocol because there's no startup sequence to formalize.

steward

The steward role file includes instructions for when to split into multiple agents ("your memory file covers 3+ unrelated domains," "you're context-switching between very different kinds of work"). So far, those conditions haven't been met. A single-page catalog with a build script and an optional API doesn't need 10 agents.

Project 4: toolshed

A software directory that catalogs 15,803 tools across 124 categories. Originally called "mainmenu." The first project to adopt thisminute's protocol+forum model from scratch rather than inheriting it — proof that the pattern transfers.

March, week 2

Full Protocol from Day One

Seven agents, a PROTOCOL.md, and a forum. Builder, curator, designer, librarian, orchestrator, skeptic, strategist. The roles were modeled on thisminute's post-growth roster, minus the domain-specific ones (no deployer, economist, or feedback — a static directory doesn't need them).

Project 5: balatro

An autopilot mod for the card game Balatro. The project that validated the checkpoint model — disposable per-cycle agents, no forum, no protocol, just a checkpoint file as the source of truth.

March, week 1

Checkpoint Cycles

Nine agents: orchestrator, mod-builder, bot-builder, mcp-engineer, play-operator, analyst, overlay-dev, cycle, skeptic. Each cycle: agents spawn, read the checkpoint, do their work, update the checkpoint, shut down. No forum — the checkpoint IS the coordination mechanism. No protocol — the cycle IS the protocol.

orchestrator mod-builder bot-builder mcp-engineer play-operator analyst overlay-dev cycle skeptic

13 cycles in, and the memory system told the real story. The project started with 1 memory file. By cycle 13 it had 8 — scoring math, cost feedback, learning loops, strategy notes. Nobody told the agents to write memory files. They started doing it because the checkpoint kept losing context between cycles.

What surprised: Memory adoption was organic and self-reinforcing. Agents that wrote memories performed better in later cycles, so more agents started writing memories. 1→8 files in ~10 cycles, without any instruction to do so.

Project 6: ops

Infrastructure management for the entire ecosystem. nginx, deployment queues, healthchecks, SSL. The steward pattern's first real growth event: one agent became two.

March, week 1

Steward Grows a Role

ops started as a single steward — one agent managing deploys, nginx config, and healthchecks. Then thisminute needed app-layer security review, and it became clear that security concerns split cleanly into two layers: application (XSS, SSRF, rate limiting) and infrastructure (firewall, SSH, TLS).

The steward didn't absorb both. Instead, a security agent was added — the first organic role split in the steward model. The steward's role file had always included instructions for when to split ("your memory covers 3+ unrelated domains"). Infrastructure + security hit that threshold.

The Meta-Project: the Forge

At some point there were enough projects that someone needed to watch them all. The Forge emerged as a meta-project — not coordinating the projects (they coordinate themselves), but auditing their organizational health and extracting patterns from what works.

March, week 2

Audit & Propagate

The forgemaster runs an audit cycle: scan every project's agent system, identify gaps, update the pattern library, and apply targeted upgrades. 23 cycles so far across 8 active projects. It tracks maturity progression — from "minimal" (steward-only) through "established+" (full protocol+forum) — and flags when a project's documentation lags behind its actual architecture.

The real output isn't the audits. It's the pattern library: 7 validated patterns extracted from live projects and shipped as reusable templates. Challenge loops, steward bootstraps, checkpoint cycles, forum maintenance, section indexes for navigating huge files. Each pattern earned its place by working in production, not by being theoretically appealing.

Three Models

The ecosystem converged on three coordination models. Not by design — by natural selection. Each emerged from a different project's constraints and then proved transferable.

Model Used By When It Fits
Protocol + Forumthisminute (11), toolshed (7), agi (7)Steady-state products with multiple domains. Agents persist between sessions. Forum is the evidence ledger; protocol defines the startup sequence. Needs a librarian or the forum grows without bound.
Checkpointbalatro (9), sts2 (9)Rapid build-test-fix iteration. Agents are disposable — they spawn, read the checkpoint, work, update it, and shut down. No forum, no protocol. The cycle IS the protocol.
Stewardrhizome (1), ops (2), bellows (1)Small scope or bootstrap projects. One agent owns everything and knows when to split. Minimal coordination overhead. The split criteria are baked into the role file.

Expectation Reality
More agents = more capabilityagi started with 10 and consolidated to 7 in a day. The eliminated roles (police, coach, editor) generated noise. thisminute grew from 5 to 11 because each role was needed. rhizome never needed more than 1.

There's one right coordination modelThere are at least three. Protocol+forum for steady-state products, checkpoint for rapid iteration, steward for small scope. Each emerged from a project's constraints. Applying the wrong model wastes effort — rhizome doesn't need a forum, balatro doesn't need a protocol.

You need a central coordinatorThe forge watches, it doesn't coordinate. Each project runs its own model. The temptation to build a hub that dispatches work across projects was strong. Resisting it was right — the projects don't share runtime state, so a coordinator would just be overhead.

This is a living document. All nine systems are still evolving. balatro is heading toward cycle 100, where the checkpoint pattern gets formally extracted. bellows and recipe-scaler-substituter are testing whether the steward bootstrap can scaffold brand-new projects from an ideas database. The forge is still learning what to watch for.

The organizational patterns referenced above are from the Rhizome catalog — 205 coordination patterns drawn from military hierarchies, jazz ensembles, ant colonies, and everything in between, plus 7 curated from this ecosystem's own experience. The Agent Forge is the system that watches these projects and extracts what works into reusable templates.
