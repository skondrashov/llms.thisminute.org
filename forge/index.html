Forge

A starter kit for giving your projects an agent system

What it is

The forge is a collection of markdown files: role definitions, workflow patterns, memory scaffolding. There is no runtime or framework. It is just context, carefully written prompt files that shape how an agent behaves in your codebase. Any harness that can read markdown files and use them as instructions will work.

The value is in saving you the cold-start problem of figuring out what good agent organization looks like. The forge gives you one opinionated answer to that, and you can take it apart and change whatever you want.

This is experimental. The whole field of multi-agent coordination is still finding its legs, and the major harness vendors will probably absorb the useful parts of this approach over time. The forge is useful now, in the gap between "agents can do real work" and "harnesses ship good defaults for organizing that work."

How to use it

Clone the repo, point your agent at AGENTS.md, and say "go." How you load the file depends on your harness — some read it automatically, others need you to pass it in (e.g., claude --system-prompt "$(cat AGENTS.md)"). On first run, the forge walks you through a setup wizard: it asks about your projects, scans what you have, designs agent structures that fit, and creates the files.

This runs an agent that reads instructions from a cloned repo

When you run the forge, your agent reads the role files in this repo as instructions. That is a supply chain risk: a compromised prompt file could instruct your agent to do something harmful. The role files are plain markdown with no obfuscation, so you can read every instruction before you run anything. Releases are tagged and include automated security scans that check for network requests, sensitive file access, encoding tricks, and confirmation bypasses.

Getting started

# Clone next to your projects
git clone --branch v0.5 https://github.com/skondrashov/agent-forge.git

# Navigate into the repo
cd agent-forge

# Point your agent at AGENTS.md and say "go"
# (aider, claude, codex, copilot, cline, cursor, goose, ...)

AGENTS.md is the entry point — tell your harness to read it. You can review the repo to see exactly what files the agent will follow as instructions.

What it does

When you point the forge at a project, it scans the codebase and sets up an agent system: an AGENTS.md that orients any future agent session, role files for different kinds of work, and a memory directory where agents can persist learnings across sessions. It draws from its own pattern library and adapts them to fit the project.

Once a project is set up, you work with it by opening an agent session in that project and talking to the orchestrator. You can ask it to build features, review code, create new roles, or answer questions. You can also ask the forge to audit your projects — it compares them against its pattern library and suggests upgrades when they'd actually help.

Roles

The forge ships with four roles that coordinate the scan-and-setup workflow. These are just markdown files that an agent reads as instructions.

Orchestrator

The coordinator. Runs the setup wizard on first use, handles audits and upgrades on request.

Auditor

Scans projects against the pattern library. Produces gap analyses and upgrade plans.

Builder

Applies upgrades to target projects. Adapts templates to fit each project's domain and conventions.

Librarian

Maintains the pattern library. Challenges the auditor's findings to keep quality honest.

Patterns

The forge ships with eleven patterns in its patterns/ directory. Some of these overlap with entries in the orchestration catalog, but the forge's versions are operational templates with concrete file trees and content you can drop into a project. A few examples:

Steward One agent, one memory file. The right default for almost every project.
Startup protocol Standalone PROTOCOL.md with timestamps, forum voting, and ref doc routing.
Shutdown reflection Agents evaluate their context at session end; feedback fixes the docs.
