Context

the window the model actually sees, and the only place anything ever gets remembered!

the model has no memory

ok so this part is genuinely wild. every time you ask the model something, it is seeing the entire conversation for the first time. it has no idea it answered you ten minutes ago. it doesn’t remember your name. it doesn’t know which tokens it wrote and which you wrote. all it has is the context window, and all it does is guess the next token that should come after everything in it.

here’s the animation from the anatomy page to show you what that looks like. hit play.

f(all preceding tokens) → one token

notice how when the model “reads” the window to predict the next token, all of the tokens flash at once. there’s no sequential pass through them, and nothing flagging which tokens it wrote last turn versus which you just sent. the whole window gets attended to simultaneously, and the next guess comes from that entire pile.

what’s filling our context?

so if the context is the only memory, the obvious next question is: what’s actually in there? like, what takes up the space?

rough token amounts for the stuff that piles up in an agent context, smallest to biggest:

system prompt
the scaffolding the model sees before anything else. role instructions, tool definitions, formatting rules. usually set once and never touched.
~1K to 5K
tool call request
when the model asks to use a tool, it writes out the function name plus a little JSON with arguments. this part is tiny, usually a sentence or two.
50 to 200
user message
whatever you typed. could be a one-word question or could be a wall of context you pasted in. usually on the smaller side!
50 to 2K
assistant reply (no thinking)
a normal model response. a paragraph, a code block, whatever. this is what you see it writing back to you.
200 to 2K
screenshot (vision)
one image, once the model converts it into tokens. a single screenshot costs about this much, regardless of what’s in it. more on this below.
~1.5K
grep hit list
the model runs a search across a repo and gets a bunch of file-line matches back. the more hits, the bigger it gets.
500 to 10K
reading a file
the model asks to open a file, and the entire file comes back to it. small config file? nothing. a 2000-line source file? ouch.
500 to 20K+
web search or scrape
the model asks a search tool, gets back a bunch of page text. scraping a single webpage can return a giant wall of stuff.
2K to 20K
reasoning trace (thinking models)
some models think out loud before answering. all of those thinking tokens live in the context too. reasoning-heavy models can burn a huge chunk of the window on a single turn.
2K to 20K+
git log -p on a big commit
the BOSS of context fillers. you ask the model to look at what changed in a commit, and suddenly fifty thousand tokens of diff show up in the window. this is the thing that eats your context window alive.
10K to 50K+

in a typical agent workflow, tool output dominates. one file read or one git diff can be bigger than everything the user and the model have typed all session combined. reasoning tokens are the other big spender when you’re running a thinking model. the system prompt, user messages, and assistant replies feel important because they’re the part you can see, but compared to tool output they’re usually small fries.

the model fills its own context, constantly

here’s the thing that makes all of this click. memory files, role instructions, skill prompts, reasoning tokens, tool output: they all look like different kinds of things, but from the model’s perspective they’re literally the same. it’s all just tokens in the window.

think about it like this. when you put a memory file in front of the model, you’re pre-populating its context. when the model writes reasoning tokens before its answer (like Gemini’s classic “but wait, let me reconsider...”), it’s populating its OWN context with words that help it think. when the harness dumps tool output into the next turn, it’s populating the model’s context on its behalf. all three are doing the same mechanical thing: putting tokens in the window so the next prediction pass has more to work with.

this is why persistent context stuff works so well and feels so intuitive. you’re not doing anything weird or magical! you’re just adding to the pile of tokens the model is about to read. the model was already going to do that to itself anyway.

vision, but make it tokens

the thing about models being pure token machines is that when they “see” an image, they’re not really looking at pixels. under the hood, a vision transformer chops the image up into little patches (usually a grid, like 16 by 16), runs each patch through an encoder, and produces a sequence of embeddings. those embeddings get stuffed into the context window right next to the text tokens, and the model treats them the same.

so when you upload a screenshot to a multimodal model, it’s not magically understanding the image, it’s receiving a sequence of image-patch tokens mixed into the same window that holds your text. typical cost: around 1,500 tokens per image in Claude, varying with model and resolution settings.

this is why document processing or long screenshot chains can chew through a context window the same way a big file read does. ten pages of scanned PDF might be 15K tokens of nothing but image patches, before you’ve said a single word to the model.

higher resolution buys more detail at the cost of more tokens. some models give you knobs for this. most just have a default patch budget and if your image needs more tokens, it downsamples.

how big is the window anyway?

this used to be a tight constraint. GPT-3 had a 4K token window, which sounds like nothing now. current frontier models:

  • 200K — Claude Haiku 4.5
  • 1M — Claude Opus/Sonnet 4.6, Gemini 2.5 Flash
  • 2M — Gemini 2.5 Pro
  • 128K — most open-weights models (Llama, Mistral, DeepSeek, Qwen)

1M is genuinely a lot! you can fit a medium-sized codebase in there. but remember, tool output is measured in the tens of thousands, so even 1M fills up faster than you’d think if the model is browsing a lot. harnesses have to decide what to evict, what to summarize, or when to start a fresh session.

and there’s a subtler thing too: even when your tokens are technically in the window, the model’s attention isn’t uniform. stuff near the beginning and the end of the window tends to be more salient than stuff in the middle. the bigger the window, the more noticeable this “lost in the middle” effect gets.
