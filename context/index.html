Context

What is context?

Every time you ask a model something, it’s seeing the entire conversation for the first time. It has no idea it answered you ten minutes ago, it doesn’t remember your name. What separates its turns from yours is the little user and assistant markers you can see in the animation below, and those are tokens too, learned during training to act as turn boundaries. Beyond those markers there’s no structural “who said what” metadata. The model has the context window, and all it does is guess the next token that should come after everything in it.

f(all preceding tokens) → one token

Notice how when the model “reads” the window to predict the next token, all the tokens flash at once. There’s no sequential pass through them. The whole window is attended to simultaneously, role markers and all, and the next guess comes out of that single pile.

What’s filling our context?

If the context is the only memory, the obvious next question is what actually lives in there, and how much space the different kinds of things take up.

Rough token amounts for the stuff that piles up in an agent context, ordered from what you’ll encounter most often to what shows up occasionally. The bars show size, not frequency, which is the interesting part: the things that happen every turn are tiny, and the things that happen sometimes are enormous.

System prompt
The scaffolding the model sees before anything else: role instructions, tool definitions, formatting rules. Always there, set once.
~1K to 5K
User message
Whatever you typed. Could be a one-word question, could be a wall of context you pasted in. Usually on the smaller side.
50 to 2K
Assistant reply
A normal model response: a paragraph, a code block, a tool call (which is just the function name plus a bit of JSON). This is the part you see it writing back to you, and it’s usually smaller than you’d think.
200 to 2K
Reasoning trace
Thinking models reason out loud before answering. Those thinking tokens live in the context too. On a hard problem, a single turn of reasoning can use more window space than the entire conversation so far.
2K to 20K+
Tool output
Everything the model gets back when it uses a tool. These all look different on the surface, but from the context window’s perspective they’re the same thing: text the harness injected because the model asked for it.
  • File contents. A targeted grep returns a few matching lines. Reading an entire file dumps the whole thing in, potentially tens of thousands of tokens.
  • Command output. A passing test suite might be 200 tokens of stdout. A failing build log might be 5K.
  • Web pages. A single documentation page or article can return more text than the entire conversation so far.
  • Search results, API responses, scraped data. Anything a tool can reach, the window has to hold.
This is the category that eats context the fastest. Whether it’s big or small depends almost entirely on whether the model already knew what it needed before it asked.
200 to 20K+
Skill injection
A reusable instruction set loaded on demand: a review checklist, a migration playbook, a coding convention guide. Mechanically identical to the system prompt, but loaded per-task rather than per-session. Slash commands, custom commands, tool descriptions with embedded examples.
500 to 5K
Screenshot
One image, converted into tokens. A single screenshot costs roughly the same amount regardless of what’s in it. More on this below.
~1.5K

In a typical agent workflow, tool output dominates. A single file read can be bigger than everything the user and the model have typed all session combined. Reasoning tokens are the other big spender. The system prompt, user messages, and assistant replies feel important because they’re the part you actually see, but next to tool output and reasoning they’re negligible. If you’ve ever worried about how many words you’re typing into a prompt, don’t!

Agents fill their own context

Memory files, role instructions, skills, reasoning tokens, tool output: they look like different kinds of things, but from the model’s perspective they’re literally the same. It’s all just tokens in the window.

When you put a memory file in front of the model, you’re pre-populating its context. When you invoke a skill (a slash command, a custom instruction set, a review checklist), the harness injects that text into the window so the model knows how to do the task. When the model writes reasoning tokens before its answer (Gemini’s classic “but wait, let me reconsider...”), it’s populating its own context with words that help it think. When the harness dumps tool output into the next turn, it’s populating the context on the model’s behalf. All four are doing the same mechanical thing: adding tokens to the window so the next prediction has more to work with.

This is why the line between roles and skills is blurry. A role is text that stays in context for the whole session. A skill is text that gets loaded for one task. Both are just tokens the model reads before predicting. The interesting orchestration question is which instructions to load when, which context belongs together in the same window, and which context needs to be split across separate agents.

Context quality matters as much as context size

It’s tempting to think of the context window as just a storage problem: will it fit? But what’s in the window affects how well the model reasons, not just whether it has room.

Too little context and the model guesses where it should know. It hallucinates function signatures, invents file paths, makes confident claims about code it hasn’t read. This is the default failure mode of a fresh conversation with no grounding.

Too much context and the model starts losing the thread. Attention is finite. The more tokens in the window, the less weight any single token gets. Important details buried in a wall of tool output or old conversation history get missed. This is why long sessions degrade: not because the window is full, but because the signal-to-noise ratio has dropped.

Irrelevant context might be the worst of the three. A window full of unrelated code, leftover conversation about a different feature, or tool output from an abandoned approach actively misleads the model. It doesn’t just waste space, it pulls reasoning off course. The model has no way to know which parts of its context are stale or off-topic.

This is also a cost question. Every token in the window is billed on input, every turn. A bloated context doesn’t just reason worse, it costs more per response. Keeping context clean is the single best lever for both quality and cost.

Vision, but make it tokens

A model is a pure token machine, so when it “sees” an image it’s not really looking at pixels. Under the hood, a vision transformer chops the image into patches (usually a 16 by 16 grid), runs each patch through an encoder, and produces a sequence of embeddings. Those embeddings get stuffed into the context window right next to the text tokens, and the model treats them the same.

When you upload a screenshot to a multimodal model, it isn’t magically understanding the image. It’s receiving a sequence of image-patch tokens mixed into the same window that holds your text. Typical cost: around 1,500 tokens per image in Claude, varying with model and resolution settings.

This is why document processing or long screenshot chains can chew through a context window the same way a big file read does. Ten pages of scanned PDF might be 15K tokens of image patches before you’ve said a single word to the model.

Higher resolution buys more detail at the cost of more tokens. Some models give you knobs for this. Most have a default patch budget and will downsample if the image needs more.

How big is the window?

This used to be a tight constraint. GPT-3 had a 4K token window, which sounds like nothing now. Current frontier models:

  • 200K  Claude Haiku 4.5
  • 1M  Claude Opus / Sonnet 4.6, Gemini 2.5 Flash
  • 2M  Gemini 2.5 Pro
  • 128K  most open-weights models (Llama, Mistral, DeepSeek, Qwen)

1M is a lot. You can fit a medium-sized codebase in there. Tool output is measured in the tens of thousands though, so even 1M fills up faster than you’d think once the model starts browsing. Harnesses have to decide what to evict, what to summarize, and when to start a fresh session.

There’s a subtler thing too: even when your tokens are technically in the window, the model’s attention across them isn’t uniform. Stuff near the beginning and the end tends to be more salient than stuff in the middle. The bigger the window, the more noticeable this “lost in the middle” effect becomes.
