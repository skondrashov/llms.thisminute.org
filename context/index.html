Context

The window the model actually sees, and the only place anything ever gets remembered.

The model has no memory

Every time you ask a model something, it’s seeing the entire conversation for the first time. It has no idea it answered you ten minutes ago, it doesn’t remember your name. What separates its turns from yours is the little user and assistant markers you can see in the animation below, and those are tokens too, learned during training to act as turn boundaries. Beyond those markers there’s no structural “who said what” metadata. The model has the context window, and all it does is guess the next token that should come after everything in it.

Notice how when the model “reads” the window to predict the next token, all the tokens flash at once. There’s no sequential pass through them. The whole window is attended to simultaneously, role markers and all, and the next guess comes out of that single pile.

What’s filling our context

If the context is the only memory, the obvious next question is what actually lives in there, and how much space the different kinds of things take up.

Rough token amounts for the stuff that piles up in an agent context, smallest to biggest:

System prompt
The scaffolding the model sees before anything else: role instructions, tool definitions, formatting rules. Usually set once and never touched.
~1K to 5K
Tool call request
When the model asks to use a tool, it writes out the function name plus a bit of JSON with arguments. Tiny, usually a sentence or two in the window.
50 to 200
User message
Whatever you typed. Could be a one-word question, could be a wall of context you pasted in. Usually on the smaller side.
50 to 2K
Assistant reply (no thinking)
A normal model response. A paragraph, a code block, whatever. This is the part you see it writing back to you.
200 to 2K
Screenshot (vision)
One image, converted into tokens. A single screenshot costs about this much regardless of what’s in it. More on this below.
~1.5K
Grep hit list
The model runs a search across a repo and gets the file-line matches back. More hits, bigger result.
500 to 10K
Reading a file
The model asks to open a file and the entire file comes back. A small config is nothing. A 2000-line source file hurts.
500 to 20K+
Web search or scrape
The model asks a search tool, gets back a pile of page text. Scraping a single webpage can return a surprising amount.
2K to 20K
Reasoning trace (thinking models)
Some models think out loud before answering. Those thinking tokens live in the context too. A reasoning-heavy model can burn a huge chunk of the window on a single turn.
2K to 20K+
git log -p on a big commit
The worst offender. You ask the model what changed in a commit and fifty thousand tokens of diff show up in the window. This is the thing that eats your context alive.
10K to 50K+

In a typical agent workflow, tool output dominates. A single file read or a single diff can be bigger than everything the user and the model have typed all session combined. Reasoning tokens are the other big spender when a thinking model is in play. The system prompt, user messages, and assistant replies feel important because they’re the part you actually see, but next to tool output they’re small.

The model fills its own context, constantly

Here’s the thing that makes all of this click. Memory files, role instructions, skill prompts, reasoning tokens, tool output: they look like different kinds of things, but from the model’s perspective they’re literally the same. It’s all just tokens in the window.

When you put a memory file in front of the model, you’re pre-populating its context. When the model writes reasoning tokens before its answer (Gemini’s classic “but wait, let me reconsider...”), it’s populating its own context with words that help it think. When the harness dumps tool output into the next turn, it’s populating the context on the model’s behalf. All three are doing the same mechanical thing: adding tokens to the window so the next prediction has more to work with.

This is why persistent context scaffolding works so well and feels intuitive. You’re not doing anything weird or magical, you’re just adding to the pile of tokens the model is about to read. The model was already going to do that to itself anyway.

Vision, but make it tokens

A model is a pure token machine, so when it “sees” an image it’s not really looking at pixels. Under the hood, a vision transformer chops the image into patches (usually a 16 by 16 grid), runs each patch through an encoder, and produces a sequence of embeddings. Those embeddings get stuffed into the context window right next to the text tokens, and the model treats them the same.

When you upload a screenshot to a multimodal model, it isn’t magically understanding the image. It’s receiving a sequence of image-patch tokens mixed into the same window that holds your text. Typical cost: around 1,500 tokens per image in Claude, varying with model and resolution settings.

This is why document processing or long screenshot chains can chew through a context window the same way a big file read does. Ten pages of scanned PDF might be 15K tokens of image patches before you’ve said a single word to the model.

Higher resolution buys more detail at the cost of more tokens. Some models give you knobs for this. Most have a default patch budget and will downsample if the image needs more.

How big is the window

This used to be a tight constraint. GPT-3 had a 4K token window, which sounds like nothing now. Current frontier models:

  • 200K  Claude Haiku 4.5
  • 1M  Claude Opus / Sonnet 4.6, Gemini 2.5 Flash
  • 2M  Gemini 2.5 Pro
  • 128K  most open-weights models (Llama, Mistral, DeepSeek, Qwen)

1M is a lot. You can fit a medium-sized codebase in there. Tool output is measured in the tens of thousands though, so even 1M fills up faster than you’d think once the model starts browsing. Harnesses have to decide what to evict, what to summarize, and when to start a fresh session.

There’s a subtler thing too: even when your tokens are technically in the window, the model’s attention across them isn’t uniform. Stuff near the beginning and the end tends to be more salient than stuff in the middle. The bigger the window, the more noticeable this “lost in the middle” effect becomes.
